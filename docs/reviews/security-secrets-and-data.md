# Security Review — Secrets, Credential Lifecycle, Sensitive Data Flow, Logging & Audit

Date: 2026-05-16
Worktree: `worktree-agent-a6a9c1e9`
Reviewer: focused secrets / data / logging / audit pass over `apps/api`, `packages/config`, `packages/contracts`, and the Prisma schema.

## 1. Executive Summary

The system enforces some good controls — strong worker-shared-secret hygiene (per-dispatch random, sha256-at-rest, single accepted dispatch), SSRF protection on outbound delivery, and listing connector secrets returns only masked metadata. Authentication uses HS256 JWTs minted from `JWT_SECRET` with PKCE on the UOA SSO flow, signed state cookies, and `__Secure-` cookie prefixes.

However, the **field name `ConnectionSecret.encryptedValue` is misleading**: the value is written and read **as plaintext** with no KMS, no field-level encryption, and no symmetric key in the env schema. There is no `ENCRYPTION_KEY` or KMS binding in `packages/config/src/env.ts`. The MCP-rule "write-only after creation, exposed back as masked metadata" is **partly honoured** on the read path but the underlying storage is plaintext, so any DB read (replica, backup, breach, logging misuse) discloses every connector secret in cleartext.

The **audit subsystem is unused**: the `AuditEvent` Prisma model exists, but `apps/` and `packages/` contain zero references to it. Every operator action (connector CRUD, secret upsert, run cancel, scenario/repo mapping CRUD, OAuth completion) and every delivery attempt silently bypasses audit despite project rules in `docs/contracts.md` and `docs/mcp-configuration.md` requiring it.

Other notable issues: the UOA OAuth callback logs the full token response (with `access_token` and any `refresh_token`) on decode failure; UOA refresh tokens are never persisted, so silent re-auth is impossible and access tokens fall over at TTL; `Connector.webhookSecret` is plaintext-at-rest in the `connectors` table; full delivery payloads and response bodies are persisted unredacted; `RuntimeApiKey` and `RuntimeApiKeyAuditEvent` are schema-only with no implementation; and the Fastify default logger silently writes OAuth `?code=`/`?state=` and the dev redirect `?token=` JWT into request logs.

## 4. Low / Hygiene

- **L-1** — `apps/api/src/services/connector-service.ts:166`: masked hint is `"*".repeat(len-4) + last4`. For very short secrets (<8 chars) this exposes a high proportion of the value. Cap the mask at a fixed width.
- **L-2** — `apps/api/src/plugins/error-handler.ts:6`: `request.log.error(error)` logs the error object. Some downstream code throws errors whose messages embed payload fragments (`status N: <body slice>` in `auth.ts:334`). Sanitize messages.
- **L-3** — `apps/api/src/routes/auth.ts:418-425`: returns the freshly-minted internal JWT, `email`, and `userId` as URL query params on the admin redirect. Browser history + referrer can leak the JWT. Prefer setting an HttpOnly cookie and redirecting to a clean path.
- **L-4** — `apps/api/src/services/dispatcher-service.ts:334`: worker shared secret uses `randomBytes(32).toString('hex')` (256 bits — fine). Verify no test fixture / seed uses a fixed secret in production seed paths.
- **L-5** — `apps/api/src/services/outbound-delivery-service.ts:33-57`: the SSRF blocklist checks hostname substring `"metadata"` / `"internal"` — easy to bypass with `169.254.169.254` directly (handled by the IP-family check) but also blocks legitimate hostnames containing those words. Use an explicit metadata-IP allowlist rather than substring.
- **L-6** — `apps/api/src/repositories/connector-repository.ts:35-43`: `deleteConnector` cascade does not delete child `repositoryMappings`, `routingRules`, `workflowScenarioBindings`. Stale references can keep secrets logically alive after the connector is "deleted".
- **L-7** — `packages/config/src/env.ts:8`: `JWT_SECRET: z.string().min(32)` — fine for HS256 length, but no rotation mechanism. Consider supporting `JWT_SECRET_PREVIOUS` for grace-window rotation.
- **L-8** — `apps/api/src/services/dispatcher-service.ts:17-21`: dispatcher writes raw `presentKeys` of the comment payload via `console.warn`. Low risk (just keys, not values) but inconsistent with the rest of the codebase which uses `app.log`.

## 5. Verified-Good

- **V-1** — `apps/api/src/plugins/worker-auth.ts:29,46-50`: incoming worker bearer is hashed with sha256 before lookup; legacy plaintext rows are upgraded on first match. Migration path is sound.
- **V-2** — `apps/api/src/services/dispatcher-service.ts:334-335,421-436` plus `apps/api/src/plugins/worker-auth.ts:67-73`: each dispatch gets a fresh 256-bit random shared secret, persisted as sha256, and the run records `acceptedDispatchAttempt`. A stale dispatch attempt is rejected with 403. This is the "short-lived, scoped, single-use" property the project rule calls for.
- **V-3** — `apps/api/src/repositories/connector-repository.ts:49-61`: `listSecrets` returns only `{id, secretType, maskedHint, createdAt, rotatedAt}` — no `encryptedValue`. The MCP write-only-after-creation read contract is implemented (note the actual stored value is plaintext per C-1, so the property only protects against the API surface, not the DB).
- **V-4** — `apps/api/src/services/outbound-delivery-service.ts:8-57`: outbound delivery does TLS scheme check, hostname allowlist, DNS resolution, and per-address RFC1918/loopback/link-local rejection. Solid SSRF posture.
- **V-5** — `apps/api/src/routes/auth.ts:17,93-100,250-256`: SSO state cookie uses `__Secure-` prefix, `HttpOnly`, `Secure`, `SameSite=lax`, path-scoped to the callback route, with HS256-signed payload + PKCE verifier. State is cleared on success and on every error path.
- **V-6** — `apps/api/src/lib/uoa.ts:82-84`: backend-to-backend bearer is `sha256(domain + secret)`, so a leaked secret cannot be replayed under a different `SSO_DOMAIN`.
- **V-7** — `apps/api/src/lib/uoa.ts:44-52`: when deriving the public JWK, private components (`d`, `p`, `q`, `dp`, `dq`, `qi`, `oth`) are explicitly stripped — defense in depth in case `exportJWK` ever returns them.
- **V-8** — `apps/api/src/services/intake-service.ts:51-59`: webhook signature is required when `connector.webhookSecret` is set; missing signature returns 401 before the body is parsed.
- **V-9** — `apps/api/src/routes/worker-api.ts:15-39`: every worker route asserts `params.jobId === request.workerDispatch.id` and that the targeted run id matches the dispatch's run id, blocking cross-run reads on a single hijacked dispatch.
- **V-10** — `apps/api/src/routes/connector-oauth.ts:39-46`: connector OAuth start verifies the connector belongs to the caller's tenant before issuing the state JWT, preventing cross-tenant connector hijack via crafted `connectorId` query params.

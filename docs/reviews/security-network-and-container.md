# Security Review - Network & Container Surface

Date: 2026-05-16
Reviewer: Claude (Opus 4.7), focused review pass
Scope: Dockerfiles, nginx, gateway WebSocket, worker subprocess sandboxing,
container start scripts, outbound HTTP / SSRF, supply-chain hygiene.
Out of scope: business-logic authz, tenancy, Prisma query safety, admin UI.

## 1. Executive summary

The repo ships a small, well-contained service split (API, gateway, worker,
admin) but the perimeter and container posture is closer to a development
prototype than an enterprise-ready deployment:

- The reverse-WebSocket gateway has no authentication, no Origin check, no
  TLS termination of its own, no message-size limit, and no rate limit. Any
  party that reaches `:3002/ws` can register as a worker and start receiving
  triage / build / merge jobs - including the per-job `workerSharedSecret`
  and full target-repo metadata. Highest-impact finding.
- Both Docker images run as root, with no `USER` directive, no pinned base
  digest, and no healthcheck. The root `Dockerfile` runs `prisma migrate
  deploy` as the CMD on every container start, sharing PID 1 with the API.
- Worker job execution uses shell-interpolated `child_process.exec` for
  `claude`, `codex`, and most `gh`/`git` invocations, with a homegrown
  `shellQuote` whose escape table omits common shell metacharacters. Any
  attacker-controllable string that reaches an `exec`'d command (prompt
  body, issue title, branch name, repo path, label name) is a potential
  injection.
- `nginx/admin.conf` is a CSR-fallback only - no HSTS, CSP, X-Frame-Options,
  Referrer-Policy, X-Content-Type-Options, rate limits, or
  `server_tokens off`. TLS is delegated to whatever runs upstream (Cloud
  Run / load balancer), so these headers must come from somewhere - today,
  from nowhere.
- The codebase has reasonable bones for hardening (worker secret hashed at
  rest, outbound delivery has a private-IP guard, webhook bodies are
  signature-parsed) but those guards have gaps and are not consistently
  applied.

The system is OK for development against trusted networks. It should not be
exposed to the public internet without addressing the Critical/High list
below.

## 4. Low / hygiene

- L-1 `Dockerfile:35`: `apt-get update && apt-get install -y openssl` in the prod stage adds CVE surface. Node 22 ships a usable OpenSSL via `node`. Verify Prisma actually needs the system openssl; if not, drop it.
- L-2 `Dockerfile.admin:1`: `node:22-alpine` for the builder + `nginx:alpine` for runtime is fine, but neither tag is pinned to a digest. Add `@sha256:...`.
- L-3 `pnpm-workspace.yaml`: no `onlyBuiltDependencies` allowlist set, no `.npmrc` with `node-linker=isolated` or `package-lock=true`. pnpm runs lifecycle scripts on install by default. For a CI image this is acceptable, but a stricter `.npmrc` with `enable-pre-post-scripts=false` for prod-image installs reduces supply-chain risk.
- L-4 `apps/gateway/src/index.ts:20-25`: `console.log` includes job IDs and queue names - fine - but the `app.log` (Fastify pino) and the bare `console.log` mix, defeating structured logs. Pick one.
- L-5 `apps/worker/src/transport/ws-transport.ts:81-95`: error-path `console.error` includes `err.message` directly - if a remote message ever contains a newline-laced string this is log injection. Use `request.log.error({ err }, 'msg')` or `JSON.stringify(err.message)`.
- L-6 `packages/github-cli/src/index.ts:248-264`: `ghCloneRepo` mutates `https://github.com/foo/bar` into `git@github.com:foo/bar`, forcing SSH. That couples the worker to a host-configured SSH key for cloning - fine in dev, undesirable in a customer's reverse-connected runtime where they may only have HTTPS + a token. Make scheme selection configurable.
- L-7 `apps/api/src/routes/auth.ts:154-181`: `/v1/auth/dev-login` is gated on `NODE_ENV !== 'production' && !UOA_CLIENT_SECRET`. The check is correct but the route is silently exposed during the entire onboarding window (before claim secret lands). Document this in `docs/sso-uoa-onboarding.md` and add a log line at startup when the route is live.
- L-8 `scripts/e2e-integration-test.mjs`: reads `LINEAR_API_KEY`, `JIRA_API_TOKEN`, etc. from env, fine. But the script also embeds hardcoded UUIDs (`ghConnectorId`, scenario IDs, label IDs) - fine for one-off testing but make the file name reflect that (`scripts/e2e-rafiki270-fixture.mjs`) and gate it behind a `--allow-real-network` flag.
- L-9 `apps/api/src/services/outbound-delivery-service.ts:42-45`: substring match on `'metadata'` / `'internal'` is fragile (matches `meta-data-cdn.com`, blocks legitimate `*-internal-foo.example.com`). Replace with explicit suffix / equality checks plus the IP-range gate, not both as string matches.

## 5. Verified-good

- API webhook ingress uses a custom JSON parser scoped via `addContentTypeParser` inside an encapsulated plugin (`apps/api/src/app.ts:107-119`) so signature verification has access to the raw body before parse. Correct pattern.
- Worker shared secrets are stored hashed (`createHash('sha256').update(secret)` in `apps/api/src/plugins/worker-auth.ts:29`) and the dispatcher stores `hashedSecret` (`apps/api/src/services/dispatcher-service.ts:425`) while the worker receives the raw secret once at dispatch time.
- Worker API client centralizes Bearer auth and never echoes the secret in URLs (`apps/worker/src/lib/api-client.ts:48-58`).
- `apps/api/src/routes/dispatcher.ts:8-13` enforces `admin`/`system` role on dispatcher endpoints, so the integration-test `dispatch-all` cannot be triggered by a non-admin JWT.
- Worker `executorFetch.url` / `skillFetches[].url` come from the API's own `buildExecutorFetchUrl`/`buildSkillFetchUrl` and are never user-supplied at the worker layer (`apps/api/src/services/dispatcher-service.ts:364-376`). The override path exists in `WorkerApiClient` but is fed from a trusted server-side construction.
- Outbound destination delivery does enforce `https:` outside development and resolves the hostname before fetch (`apps/api/src/services/outbound-delivery-service.ts:33-57`). The implementation is partial (see H-7) but the intent is right.
- `parseEnv` requires `JWT_SECRET` to be at least 32 chars and validates `DATABASE_URL` / `API_BASE_URL` as URLs (`packages/config/src/env.ts:3-58`).
- The worker's `codex-exec` utility uses `spawn(..., { shell: false })` with an argv array (`apps/worker/src/utils/codex-exec.ts:22-26`) - the correct pattern that the rest of the worker should adopt (see H-5).
- `apps/api/src/routes/connector-oauth.ts:43-46` signs the OAuth state JWT with a 10-minute expiry and verifies `platformKey` round-trip on callback.

## Suggested ordering for fixes

1. H-1 (gateway WS auth) - fix this before any non-trivial deploy.
2. H-5 (worker shell injection) - replace shell-interpolated `exec` calls + `shellQuote` with `execFile`/`spawn`. Mechanical refactor with high payoff.
3. H-3, H-4 (Dockerfile non-root, signal handling, separate migrate step).
4. H-6 (nginx headers) - five lines of config, immediate XSS/clickjacking benefit.
5. H-7 (SSRF TOCTOU) - undici Agent with pinned IP.
6. H-2, M-2, M-3 (WS payload limits, TLS, heartbeat liveness).
7. M-7 (worker sandboxing) - needs a real design pass; this is the largest piece of long-running work.

## Resolution notes

Date: 2026-05-16
Branch: `fix/security-input-network-2026-05-16`

- **H-6 — nginx security headers** (`nginx/admin.conf`, `Dockerfile.admin`): the admin server block now emits HSTS, X-Content-Type-Options, X-Frame-Options (`DENY`), Referrer-Policy (`no-referrer`), Permissions-Policy, and a strict Content-Security-Policy (`default-src 'self'`, `frame-ancestors 'none'`, `object-src 'none'`, etc.) on every response. `server_tokens off;` is set. CSP `connect-src` includes a templated `${ADMIN_API_CONNECT_SRC}` placeholder which the Dockerfile sets to `'self'` by default; deployers override the env var with their API origin (e.g. `https://api.appbuildbox.com`). gzip is on for text/asset types. Long-cache `Cache-Control: public, max-age=31536000, immutable` is set on `/assets/` and on hashed static files; `index.html` is served `no-cache`. nginx's built-in template substitution (`/etc/nginx/templates/*.template`) performs the env substitution at boot, so no entrypoint script is needed.

- **H-7 — SSRF TOCTOU + substring filter** (`apps/api/src/services/outbound-delivery-service.ts`, `packages/contracts/src/safe-fetch.ts`): the old `validateOutboundUrl` (resolve, then fetch — two independent DNS calls) was deleted. The outbound delivery service now calls `safeFetchFollowRedirects` from the shared `packages/contracts/src/safe-fetch.ts` helper. That helper resolves the hostname once, validates every resolved address against an explicit CIDR blocklist (`0.0.0.0/8`, `10/8`, `100.64/10`, `127/8`, `169.254/16`, `172.16/12`, `192.168/16`, `198.18/15`, multicast, ULA, link-local v6, IPv4-mapped v6), then constructs an undici `Agent` whose socket-layer `lookup` returns ONLY one of the pre-validated addresses. The fetch can therefore never land on an unvalidated IP, even if the OS resolver flips. Redirects are not followed at the dispatcher level; `safeFetchFollowRedirects` walks each `Location` hop and re-runs the full validation on each. Tests in `packages/contracts/src/safe-fetch.test.ts` cover IPv4 / IPv6 private ranges, IPv4-mapped-IPv6 metadata bypass, percent-encoded URLs, missing-scheme rejection, host allowlists, and a DNS-rebind simulation where the redirect target resolves to `169.254.169.254` and must be refused.

- **Single source of truth.** The same `safeFetchFollowRedirects` is the only SSRF guard in the repo. The Jira and Respond.io clients (M-1 in the input/injection review) call it with a per-platform host allowlist. The old per-service substring filter on "metadata"/"internal" is gone.
